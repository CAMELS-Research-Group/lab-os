# Cohort Retrospective — Facilitator Runbook

**Status:** draft
**Audience:** Facilitators running the optional **cohort retrospective** — the collective close that
turns a live cohort's shared experience into written conventions. Read alongside the participant page
[`site/docs/workshops/closeout/cohort-retrospective.md`](../../../site/docs/workshops/closeout/cohort-retrospective.md)
and the main [Closeout facilitator runbook](./facilitator-runbook.md). The participant page frames the
*why*; this runbook is the *run-of-show*.

---

## How it fits

The main closeout is **individual**: each participant builds a carry-forward artifact for *their* next
project, and the collective dimension is the presentation round. The cohort retrospective adds the
missing collective step — a single facilitated session that turns the group's shared two weeks into a
few **named conventions the team adopts**, written into `.claude/rules/`, a `CLAUDE.md`, or
`project_log.md`.

Run it **after** the individual carry-forward and presentations (Movements 2–4 of the main runbook),
while the experience is fresh. It is **optional** and **for live cohorts or teams** — skip it for
solo, self-paced readers. Budget **~90 minutes**, a group of **≤8** in one room, a wall, sticky
notes, and dot stickers.

> **You do not need to be a retro expert.** Every block below has a timebox, what you do, and the
> exact words to say (in quotes — read them aloud). The two things that make it work: protect the
> **silent solo write** (Block 2) so quiet voices reach the wall, and do not shrink **Codify**
> (Block 5) — that block is what produces conventions instead of sticky notes.

## Why this shape (the formats)

Three formats are layered, each chosen for a one-time milestone close that must extract durable
patterns — not a recurring sprint retro:

1. **4Ls — Liked / Learned / Lacked / Longed-for** *(the harvest)*. A milestone-fit reflection format
   whose **Learned** and **Longed-for** lenses pull out the *why behind the what* — the tacit
   knowledge the close exists to surface.
2. **Keep / Drop / Try** *(the output spine)*. Forces each reflective note into something you can act
   on: keep = a convention to write down, drop = a habit to stop, try = something new to pilot.
3. **1-2-4-All + dot voting** *(the engine)*. Silent solo → pairs → small groups → whole group gets
   every person's experience onto the wall (no loudest-voice capture), then dot voting converges to
   the few patterns worth codifying.

The flow: 1-2-4-All sets *how* people contribute → 4Ls sets *what* they reflect on → Keep/Drop/Try
sets the *output* → dot-vote + **Codify** turns the output into named, owned conventions.

## Before the session (5-minute prep)

- [ ] A **wall** with four labeled columns: **Liked · Learned · Lacked · Longed-for**.
- [ ] **Sticky notes** (one idea per sticky) + markers for everyone; **5 dot stickers** per person.
- [ ] A second surface with three columns: **Keep · Drop · Try**.
- [ ] The **scribe's laptop** open to the capture template below, on a shared screen if you have one.
- [ ] Print this page; optionally print Appendix C (anti-pattern bank) and Appendix D (pre-seeded
      patterns) as backstops.
- [ ] Two roles assigned (see below). At ≤8, Facilitator + Scribe is enough.

## Roles

- **Facilitator** — runs time and reads the prompts; stays *neutral on content* (contributes ideas
  only in writing, like everyone else); enforces the silent write and protects the Codify block.
- **Scribe** — fills the capture template live as the group talks; owns getting the §3 codify table
  complete (name · one-line rule · home · owner) before anyone leaves.

## Run-of-show (90 min)

Programmed: 84 min + ~6 min slack. Each block: **goal · time · what you do · what you say.**

### Block 1 — Frame & safety (8 min)

**Goal:** everyone knows the output is *conventions, not blame*, and how the session runs.

**You do:** welcome, state the goal, set the norm, walk the wall, name the two roles.

**You say:**
- *"We spent two weeks building with AI. The next 90 minutes turn what we learned into conventions
  the next cohort inherits — written rules, not vibes. By the end we'll have things to keep, drop,
  and try, and for the ones worth keeping, where each gets written down and who writes it."*
- *(Norm)* "Everyone did the best they could with what they knew at the time. We're examining the
  work and the workflow, not the people. What's said here stays here and is only for improving how
  we work."
- *"Four columns on the wall — Liked, Learned, Lacked, Longed-for. You'll write silently first, then
  pair up, then we cluster and decide. \<Scribe> captures as we go."*

### Block 2 — Solo recall + 4Ls write (18 min) — *the "1" and "2"*

**Goal:** get every person's honest material onto stickies before any discussion.

**You do:** run a 3-min silent recall, then a 10-min silent write, then a 5-min pair merge. Keep it
quiet — resist filling the silence.

**You say:**
- *(3 min recall)* "Silently walk back through the two weeks — planning, building, the close. Jot a
  private timeline: the moments the AI made something click, and the moments it cost you time."
- *(10 min write — one idea per sticky; read the four lenses)*
  - **Liked:** *"What worked — about how we used the AI, the specs, the workflow — that you'd want
    again?"*
  - **Learned:** *"What did you learn about working **with** the AI that you didn't know two weeks ago
    — when to trust it, when to verify, how to hand it work, how to write a spec it could follow?"*
  - **Lacked:** *"What was missing — a habit, a guardrail, a piece of the spec, a check — that would
    have saved you?"*
  - **Longed-for:** *"What do you wish existed — a tool, a rule, a template, a step in the workflow?"*
  - *"Stuck? Grab any question from the prompt bank (Appendix B)."*
- *(5 min pairs)* "Turn to one neighbor. Compare notes, merge duplicates, and sharpen the wording so
  a stranger would understand each sticky. Add any new ones the conversation sparks."

### Block 3 — Small groups + surface to wall (14 min) — *the "4" and "All"*

**Goal:** converge to the notes that matter and get them clustered.

**You do:** pairs join into fours (7 min), then read the wall aloud and cluster with the Scribe (7 min).

**You say:**
- *(7 min fours)* "Join another pair. As a four, pick the notes that matter most or that more than one
  of you wrote — not all of them, your strongest. Post each under the right column."
- *(7 min read + cluster)* "I'll read the wall. Call out if two of these are really the same thing, or
  if one's in the wrong column." *(You and the Scribe group stickies into named themes.)*

### Block 4 — Converge → Keep / Drop / Try + dot vote (16 min)

**Goal:** every theme becomes a Keep/Drop/Try action, ranked by what's worth codifying.

**You do:** walk each cluster and sort it (10 min); run the dot vote (6 min).

**You say:**
- *(10 min sort)* "For this theme — should we **Keep** doing it, **Drop** it, or **Try** something
  new? Say it as an action: 'Keep: …', 'Drop: …', 'Try: …'." *(If the Drop column is thin, seed it
  from Appendix C:)* "Here are some common AI-dev traps — did any of these bite us?"
- *(6 min vote)* "Five dots each. Put them on the keep/drop/try items most worth writing into our
  conventions — what a new teammate most needs to inherit. Stack on one if you feel strongly."
  *(Rank by dots; carry the top ~5–7 into Block 5.)*

### Block 5 — Codify: name + route the patterns (20 min) — *the block that matters*

**Goal:** turn the top-voted items into **named conventions with a home and an owner**.

**You do:** for each top item, run the group through three moves while the Scribe fills §3 live:
**name it → state it as one line → route it to a home + owner.**

**You say:**
- *(name)* "Give it a short handle you could say out loud — like 'spec-before-keys', 'read-the-diff',
  'verify-before-claim'."
- *(one-line rule)* "Write it as a rule a new teammate could follow tomorrow — one sentence."
- *(route — type)* "Is this a **rule for everyone** (`.claude/rules/`), a **fact about how we work**
  (a `CLAUDE.md`), or a **one-time decision worth remembering** (a `project_log.md` entry)?"
- *(route — scope)* "Is it for **every** project (a top-level rule) or just **this** repo (a per-repo
  rule / that repo's `CLAUDE.md`)?"
- *(owner)* "Who owns turning this into a pull request?"
- *(backstop, if the group stalls or has time)* "Here's a starter list of practices this workflow
  already assumes (Appendix D) — which did we actually live by, and which should become explicit?"

### Block 6 — Close: owners, follow-up, thanks (8 min)

**Goal:** nothing leaves the room un-owned; end on a human note.

**You do:** sweep the codify table for gaps, agree the follow-up, save the artifact, run appreciations.

**You say:**
- *"Every row in the codify table — does it have a name, a home, and an owner? If not, we assign it
  now or it goes to the parking lot."*
- *"One follow-up: a single pull request (or a small few) that writes these into their homes. Who
  opens it, and by when?"*
- *(Scribe saves the capture artifact.)*
- *(appreciation round)* "One sentence — something a teammate did these two weeks worth calling out."

## Capture template

*The Scribe fills this live. Keep §1 brief (it's source material); spend the care on §3.*

```markdown
# <Cohort/Team> — Retrospective Capture · <DATE>

**Participants:** <names or count>  ·  **Facilitator:** <name>  ·  **Scribe:** <name>

## §1 — Raw harvest (4Ls clusters)
Brief — the themes off the wall, not every sticky.
- **Liked:** <theme> · <theme>
- **Learned:** <theme> · <theme>
- **Lacked:** <theme> · <theme>
- **Longed-for:** <theme> · <theme>

## §2 — Keep / Drop / Try (dot-ranked)
| Item (as an action) | K/D/T | Votes | One-line rationale |
|---|---|---|---|
| Keep: <…> | K | <n> | <why it earned votes> |
| Drop: <…> | D | <n> | <…> |
| Try: <…> | T | <n> | <…> |

## §3 — Patterns & habits to codify  ← the deliverable
| Pattern name | One-line rule | Evidence from the two weeks | Type | Home | Owner | Status |
|---|---|---|---|---|---|---|
| <handle> | <rule a newcomer could follow> | <when it showed up> | convention / guardrail / pilot | <see routing aid> | <name> | proposed |

## §4 — Action items
| Owner | Action | Lands in (PR / file) |
|---|---|---|
| <name> | Open the conventions PR writing §3 rows into their homes | <PR#> |

## §5 — Parking lot
Out-of-scope or unresolved — <…>
```

**Routing aid — pick the home before anyone leaves:**

| If the pattern is… | Home |
|---|---|
| A rule everyone should follow, every project | a top-level rule in `.claude/rules/` |
| A rule for one project | that repo's `.claude/rules/` (per-repo rule) |
| A fact about how the team or a repo works | the relevant `CLAUDE.md` |
| A one-time decision worth remembering | a `project_log.md` entry |

See [Rules, explained](../../../site/docs/rules-explained.md) for what belongs in a rule file vs a
`CLAUDE.md`, and the [`project_log.md` template](../../../templates/project_log.template.md) for the
log-entry shape. A rule or `CLAUDE.md` change is a documentation change — open it as a normal pull
request so it gets reviewed before it becomes "how we work."

## Facilitation do / don't

**Do:**
- Protect the silent solo write in Block 2. It is what gets every voice onto the wall instead of the
  two loudest. If people start talking, gently restart the quiet.
- Keep the Codify block (5) sacred. If you are running behind, cut slack or trim Block 1 — never
  Codify. A retro that names patterns but routes none of them produces nothing durable.
- Make people *name* each kept pattern. A handle a teammate can say out loud ("read-the-diff") sticks;
  a paragraph does not.
- End every codify row with a home and an owner before the room empties. An un-owned convention is a
  good intention, not a change.
- Let people be honest about what did not work with the AI. The drop column is only useful if it is
  truthful; a session where everyone reports the AI was great produces no guardrails.

**Don't:**
- Let it become an open discussion from the start. The structure (write → pair → four → all) is what
  prevents groupthink and the loudest-voice problem. Run the structure.
- Read Appendix D (pre-seeded patterns) early. It anchors the group and crowds out their own patterns.
  It is a backstop for a stalled or tired room, not an opening move.
- Try to codify everything. Dot voting exists so you write down the few that matter. Five to seven
  codified conventions from a 90-minute session is a strong outcome.
- Confuse this with the individual carry-forward. That artifact is each person's private input to
  their next project; this session produces the team's shared conventions. Keep them distinct.

## Common questions from participants

**"Isn't this the same as the carry-forward worksheet?"** No. The carry-forward artifact is
individual and travels to *your* next project. The cohort retrospective is collective and changes how
*the team* works — its output is written conventions everyone adopts.

**"What if we can't agree where a pattern lives?"** Park the routing, not the pattern. Record the
name and one-line rule in §3, mark the home "TBD", and let the owner decide in the follow-up pull
request where it best fits. Don't burn live time on filing decisions.

**"What if nothing reaches five to seven patterns?"** Fewer is fine — two well-named, well-routed
conventions beat seven vague ones. Use Appendix D only if the room genuinely runs dry, and only to
react to, not to copy.

**"We're remote / hybrid — does this still work?"** Yes, with a shared digital board (one column per
L, a Keep/Drop/Try area, and a voting mechanism) as the single source of truth, and breakout rooms
for the pair and small-group steps. The block structure and prompts are unchanged.

|---|---|---|---|
| 1 | Frame & safety | 8 | Goal = conventions not blame · norm · walk the wall |
| 2 | Solo recall + 4Ls write | 18 | 3 silent recall → 10 silent sticky-write → 5 pair merge |
| 3 | Small groups + surface | 14 | 7 fours post strongest → 7 read wall + cluster |
| 4 | Keep/Drop/Try + vote | 16 | 10 sort clusters into K/D/T → 6 dot-vote (5 each), rank |

## Appendix B — AI-collab prompt bank

Use any of these in Block 2 if someone is stuck.

- **Trust & verification:** When did you trust AI output and get burned? When did you verify and catch
  something? What's your rule now for what gets read line-by-line vs skimmed?
- **Spec quality:** When did a vague spec produce slop? When did *tightening the spec* fix the output
  instead of re-prompting? Did you re-prompt when you should have fixed the spec or plan?
- **Handoff & scope:** What size/kind of task did the AI handle well vs badly? When did you
  over-delegate a judgment call? When did you do by hand what it could have done?
- **Speed vs control:** Where did AI genuinely save hours? Where did it *feel* fast but cost more in
  review and cleanup?
- **Guardrails:** Which guardrail (a verification gate, a review, a checklist) actually caught
  something? Which felt like ceremony?
- **Workflow fit:** Where did spec → plan → build → verify help? Where did it get in the way, and what
  would you change?

## Appendix C — Anti-pattern "drop" bank

Read these in Block 4 if the Drop column is thin ("did any of these bite us?").

- **Rubber-stamp merge** — accepting AI output (or an AI review) without reading the diff.
- **Re-prompt spiral** — re-prompting the model instead of fixing the spec or plan it works from.
- **Spec skip** — going straight to code with the AI, no plan.
- **Gate-by-pipe** — running the verification command piped so its exit code is swallowed and a red
  gate reads green.
- **Context dump** — pasting huge undirected context instead of a tight, scoped ask.
- **Silent plan drift** — letting the AI deviate from the agreed plan without noting it.
- **Confident-wrong** — trusting an authoritative-sounding answer with no source or verification.
- **Over-delegation** — handing the AI a call that needed a human.
- **Slop accretion** — keeping AI-generated code or docs nobody on the team fully understands.
- **Token / time burn** — long autonomous loops with no cap and no checkpoint.

## Appendix D — Pre-seeded candidate patterns (reaction set — anchoring risk)

Practices this workflow already assumes. Use **only** as a Block-5 backstop ("which did we live by;
which should be explicit?"). Don't read it early — it anchors the group. Each is a candidate **Keep**
unless the group disagrees.

- **Spec-before-keys** — no non-trivial build without a plan (problem, scope, tasks, verification).
- **Code-free plans** — plans say *what* to satisfy, not *how*; the behaviors are the test surface.
- **Verify-before-claim** — run the verification gate and read the output before calling something
  done.
- **Read-the-diff** — a human reads AI output before it merges.
- **Right-sized delegation** — hand the AI tasks with a clear done-condition; keep judgment calls and
  ambiguous scope hands-on.
- **One decision, one home** — write a durable decision down once (a rule, a `CLAUDE.md`, or
  `project_log.md`) and link to it rather than restating it.
